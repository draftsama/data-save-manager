<objective>
Protect DSMSlot.Set/Get against type mismatches using the existing DSMConstant codegen metadata as the single source of truth (SCHM-01), gated by a new DSMConfig.StrictSchema flag that defaults to lenient warn-and-coerce (SCHM-02).

Purpose: A typo'd value type (e.g. Set("testKey", "abc") where testKey is int) must never silently corrupt a save. Lenient mode warns and coerces so upgrades don't break existing call sites; strict mode throws for teams that want hard failures. The schema is derived by reflecting the SAME DSMConstant fields DSMSlot.SeedDefaults already reads, so key types are never maintained twice.

Output: New Runtime/DSMSchema.cs (schema + DSMSchemaViolationException), a DSMConfig.StrictSchema flag, validation wired into DSMSlot.Set/Get, and an EditMode test fixture defining the behavior.
</objective>

## Phase Goal (MVP)

**As a** game developer integrating DSM, **I want to** have Set/Get reject or coerce values whose type does not match a key's declared schema type, **so that** a mistaken value type never silently corrupts a save file.

Note: derived from the ROADMAP Phase 3 `**Goal:**` line. DSM is a library, so its "user" is the integrating developer; the ROADMAP goal was not authored in user-story form and is restated here without changing scope.

## Phase Goal

**As a** game developer using DSM, **I want to** keep the save system's shared state and save/load pipeline safe under concurrent access and invalid input, **so that** my players never lose or corrupt their save data.

## Acceptance Criteria (Slice 2)
- [ ] Saves write to a temp path then atomically replace the destination; no `.tmp` file survives a successful save.
- [ ] A rapid `Set()` storm never throws `ObjectDisposedException`; the final value is persisted.
- [ ] `Load`/`Save`/`LoadAsync`/`SaveAsync` are serialized by a per-slot `SemaphoreSlim`, always released in `finally`.
- [ ] Concurrent `Set`+`SaveAsync`, `Set`+`Load`, and multi-watcher tests pass with final-state assertions.

|----------|-------------|

| main thread ↔ background save task | debounce loop + `SaveAsync` run off the calling thread and share `_data` and the save path |

| T-01-04 | Denial of Service | Disposed-`CancellationTokenSource` `ObjectDisposedException` under rapid `Set()` | high | mitigate | Single long-lived debounce loop (Task 2, D-01) removes the per-call CTS entirely; storm test asserts no throw |

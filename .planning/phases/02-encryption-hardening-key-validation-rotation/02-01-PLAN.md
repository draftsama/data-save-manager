<objective>
Deliver tamper-evident encrypted saves and one shared encryption-key validator. This plan rewrites `DSMEncryptor` from the current AES-CBC-only (no integrity) scheme to Encrypt-then-MAC (AES-CBC + HMAC-SHA256), and introduces `DSMEncryptionKey.Validate` as the single accessor every runtime and Editor code path uses to reject empty/short keys.

Purpose: Encrypted saves today are confidential but NOT tamper-evident — a modified or truncated `.enc` file decrypts to corrupted data (or throws an opaque padding error) instead of being detected. Key validation is also duplicated/absent, so an empty key silently produces a weakly-derived cipher. This plan closes both gaps (ENC-01, BUGS-02) and lands the encryption-edge-case tests (TEST-02: wrong key, truncated file, empty key).

Output: A rewritten `DSMEncryptor`, a new `DSMEncryptionKey` validator, validation wired into `DSMConfig.SetEncryptionKey`, and a passing edge-case test suite.
</objective>

## Phase Goal (MVP user story)

**As a** game developer using DSM, **I want to** have encryption keys validated everywhere and encrypted saves that detect tampering, **so that** a corrupted or modified save file is caught with a clear error instead of loading garbage into my game.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| disk file -> DSM decrypt | An `.enc` file on disk is untrusted input; it may have been modified, truncated, or replaced before load |
| developer/game code -> DSMConfig.SetEncryptionKey | Key material enters the system here and via DSMEncryptor.Encrypt/Decrypt; must be validated at the boundary |
| Editor tooling -> DSMEncryptor | DSMConfigEditor and DSMManagerWindow call the same Encrypt/Decrypt chokepoint as the runtime |

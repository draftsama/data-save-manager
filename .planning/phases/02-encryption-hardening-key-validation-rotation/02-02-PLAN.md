<objective>
Deliver atomic encryption-key rotation. This plan adds `DSM.RotateEncryptionKeyAsync(newKey)`, which decrypts every encrypted slot with the current key, re-encrypts it with the new key using the Encrypt-then-MAC format from Plan 02-01, and commits the new key to `DSMConfig` only after all slots succeed — with a journal so an interrupted rotation is recoverable rather than leaving slots in a mixed or unreadable state.

Purpose: Rotating an encryption key today is impossible — there is no API (STATE.md/observation confirms none exists), so a leaked or weak key can never be replaced without hand-editing files. A naive per-slot re-encrypt would risk partial application: some slots on the new key, some on the old, and a single committed config key that can only read one set. This plan makes rotation all-or-nothing at the staging boundary and recoverable at the commit boundary (ENC-02), and lands the key-change-between-saves test (TEST-02).

Output: `DSM.RotateEncryptionKeyAsync`, `DSMSlotManager.RotateEncryptionKeyAsync` with startup journal recovery, per-slot stage/commit re-encrypt helpers reusing the Phase 1 `_ioGate` + `ReplaceFile`, and a passing rotation test suite.
</objective>

## Phase Goal (MVP user story)

**As a** game developer using DSM, **I want to** rotate the encryption key with a single call that never loses or corrupts a slot, **so that** I can respond to a compromised or weak key without risking my players' save data.

> This slice depends on Plan 02-01 (it re-encrypts using the Encrypt-then-MAC format and the shared DSMEncryptionKey.Validate accessor introduced there). See 02-01 for the ROADMAP-goal-vs-user-story note.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| game code -> DSM.RotateEncryptionKeyAsync | The new key enters here; must be validated before any file mutation |
| in-flight rotation -> disk | Multiple slot files + one config key must transition together; a crash may occur between file operations |
| fresh DSMSlotManager -> disk journal | On construction, a leftover journal indicates an interrupted rotation to be recovered before any load |
| concurrent Set/SaveAsync -> slot files | Autosave/debounce may fire during rotation and race the re-encrypt |
